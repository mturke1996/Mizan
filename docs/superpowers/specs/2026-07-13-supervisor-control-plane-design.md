# مركز تحكم المدير الكامل — مواصفة معتمدة

التاريخ: 2026-07-13  
الحالة: معتمد من المستخدم للتخطيط والتنفيذ

## 1. الهدف

تحويل `/supervisor` من لوحة متابعة محدودة إلى مركز تشغيل كامل لإدارة:

- العملاء المستقلين ومساحات عملهم.
- الحسابات وحالات الوصول.
- دورة الاشتراك والتجديد والتجميد والإلغاء.
- خطط الاشتراك والأسعار والمزايا.
- المدفوعات المعلقة وسجل المراجعات.
- الرسائل والإشعارات داخل التطبيق.
- التحليلات التشغيلية والفوترة.
- سجل تدقيق غير قابل للإخفاء.
- الاطلاع الدائم، للقراءة فقط، على بيانات العميل المالية.

## 2. القرارات المثبتة

- إنشاء العملاء بطريقتين:
  1. دعوة آمنة بالبريد.
  2. إنشاء مباشر بكلمة مرور مؤقتة.
- كلمة المرور المؤقتة يمكن عرضها مرة واحدة أو استبدالها برابط تعيين عبر البريد.
- كل مستخدم ينشئه المدير عميل مستقل بملف ومساحة واشتراك خاص.
- الاشتراكات هجينة: طلبات دفع وموافقة، مع تحكم يدوي كامل للمدير.
- المدير يستطيع إنشاء وتعديل وأرشفة الخطط والأسعار والمدد والمزايا.
- الرسائل داخل التطبيق في الإصدار الحالي، مع قابلية إضافة البريد لاحقًا.
- لا حذف صامت للعملاء أو الخطط؛ التعطيل والأرشفة هما المسار الطبيعي.
- وصول المدير للدفاتر المالية دائم لكنه قراءة فقط.
- كل قرار أو فتح لبيانات مالية حساسة يُسجل في سجل التدقيق.

## 3. حدود الخصوصية والأمان

- مفتاح `service_role` لا يصل إلى React أو أي حزمة عامة.
- عمليات Supabase Auth الإدارية تنفذ داخل Edge Function فقط.
- Edge Function تتحقق من JWT، ومن أن `profiles.system_role = supervisor` و`account_status = active`.
- كل RPC إداري يتحقق داخليًا من `private.is_supervisor()`.
- Views العامة الجديدة تكون `security_invoker` أو تُستبدل بـ RPCs مقيّدة.
- الوصول المالي للمدير لا يتيح الإدخال أو التعديل أو الحذف.
- إيقاف العميل لا يحذف سجلاته المالية أو سجل اشتراكه.
- أرشفة الخطة لا تغيّر الاشتراكات التاريخية المرتبطة بها.
- الإجراءات الحساسة تتطلب ملاحظة إدارية وتخزن الحالة السابقة واللاحقة.

## 4. البنية التقنية

### 4.1 Edge Function

اسم مقترح: `supervisor-customer-admin`

الإجراءات:

- `invite_customer`
- `create_customer_with_password`
- `send_password_setup`

المدخلات المشتركة:

- البريد.
- الاسم.
- اسم مساحة العمل.
- العملة.
- طريقة الإنشاء.
- الخطة الأولية.
- حالة الاشتراك والمدد.
- سبب الإجراء.

قبل استدعاء Auth Admin API، تنشئ Edge Function «نية إعداد عميل» قصيرة العمر في جدول خاص عبر RPC يعمل بهوية المدير. تحتوي النية على البريد الموحّد، إعداد المساحة، الخطة، المدة، وطريقة كلمة المرور، ولا تحتوي كلمة المرور نفسها.

يستهلك trigger إنشاء المستخدم هذه النية داخل معاملة إنشاء Auth user نفسها، ثم ينشئ:

- `profiles`
- `workspaces`
- owner membership
- الاشتراك والخطة والمدد المحددة، أو trial افتراضي عند التسجيل العام
- المحفظة والحسابات الأساسية
- الإشعار وسجل التدقيق

إذا فشل أي جزء من bootstrap يفشل إدخال Auth user كله ولا يبقى عميل نصف مكتمل. إذا فشل استدعاء Auth قبل إنشاء المستخدم، تلغي Edge Function النية، كما تنتهي النوايا غير المستهلكة تلقائيًا بعد مدة قصيرة.

### 4.2 تغييرات قاعدة البيانات

ترحيل مستقل باسم وصفي قريب من:

`supervisor_control_plane`

إضافات `profiles`:

- `must_change_password boolean not null default false`

جدول خاص غير معروض في Data API:

- `private.customer_onboarding_intents`
- مفتاحه البريد الموحّد، مع expiry وبيانات الإعداد و`created_by`
- لا يخزن كلمات مرور أو أسرار
- يُستهلك مرة واحدة فقط داخل `private.handle_new_user`

RPCs إدارية:

- `supervisor_create_plan`
- `supervisor_update_plan`
- `supervisor_archive_plan`
- `supervisor_change_subscription_plan`
- `supervisor_renew_subscription`
- `supervisor_set_subscription_state`
- `supervisor_set_subscription_dates`
- `supervisor_send_notification`
- `supervisor_prepare_customer_onboarding`
- `supervisor_cancel_customer_onboarding`
- `complete_required_password_change`

RPCs قراءة مالية مقيّدة:

- `supervisor_customer_financial_snapshot`
- `supervisor_customer_wallets`
- `supervisor_customer_transactions`
- `supervisor_customer_projects`
- `supervisor_customer_workers`

جميع RPCs التفصيلية تدعم pagination وحدودًا صريحة.

Views/queries تشغيلية:

- قائمة العملاء الموحّدة مع الحساب والمساحة والاشتراك والخطة.
- تاريخ المدفوعات مع الخطة والمالك والمراجع.
- مؤشرات الإيراد حسب العملة والفترة.
- الاشتراكات القريبة من الانتهاء أو في المهلة.
- سجل النشاط مثرى باسم العميل والمنفذ.

## 5. دورة الاشتراك

الحالات المدعومة:

- `trialing`
- `active`
- `grace`
- `frozen`
- `expired`
- `cancelled`

الإجراءات:

- تغيير الخطة.
- تجديد فترة أو عدة فترات.
- تعيين تاريخ البداية والنهاية.
- منح أيام مجانية.
- إدخال أو إنهاء المهلة.
- تجميد وإلغاء التجميد.
- إلغاء فوري أو بنهاية الفترة.
- إعادة تفعيل.

كل انتقال يتحقق من صحته في قاعدة البيانات ويكتب `subscription_events`.

## 6. خطط الاشتراك

حقول المحرر:

- code ثابت وفريد.
- الاسم.
- السعر والعملة.
- دورة الفوترة وعدد الوحدات.
- أيام التجربة.
- عامة أو خاصة.
- نشطة أو مؤرشفة.
- مزايا JSON منظّمة من مفاتيح معروفة.

لا تعديل رجعي لسعر طلب دفع قائم. طلب الدفع يحتفظ بقيمته التي أُنشئ بها.

## 7. الرسائل

الإصدار الحالي يدعم إشعارات داخل التطبيق:

- إلى عميل واحد.
- إلى مساحة محددة.
- إلى كل العملاء النشطين.
- إلى شرائح: تجريبي، قريب الانتهاء، مجمّد، أو متأخر.

الحقول:

- العنوان.
- النص.
- النوع.
- المستهدف.
- metadata.
- المرسل ووقت الإرسال.
- حالة القراءة من جدول notifications الحالي.

لا بريد تسويقي في هذه المرحلة.

## 8. الوصول المالي للمدير

يعرض تبويب «البيانات المالية — قراءة فقط»:

- إجمالي الرصيد حسب العملة.
- المحافظ.
- المعاملات مع pagination.
- المشاريع ومؤشراتها.
- العمال والمستحقات.

القواعد:

- لا أزرار تعديل أو حذف.
- شارة ثابتة «قراءة فقط».
- كل فتح يسجل `supervisor.financial_accessed`.
- لا يتم تحميل التفاصيل قبل فتح التبويب.
- لا تجمع العملات المختلفة في رقم واحد.

## 9. بنية الواجهة

### 9.1 التنقل

المجموعات:

1. التشغيل
   - مركز العمليات
   - المهام والتنبيهات
2. العملاء
   - العملاء
   - الاشتراكات
3. الفوترة
   - المدفوعات
   - الخطط والأسعار
   - الإيرادات
4. التواصل
   - الرسائل
5. الحوكمة
   - سجل التدقيق
   - الأمان

### 9.2 مركز العمليات

ترتيب المحتوى:

1. طابور قرارات قابل للتنفيذ.
2. المدفوعات المعلقة وقيمتها حسب العملة.
3. الاشتراكات المنتهية قريبًا والمهلة والتجميد.
4. الإيراد ومعدل الموافقة ومتوسط زمن المراجعة.
5. توزيع الخطط وتحويل التجارب.
6. آخر القرارات الإدارية.

لا مؤشرات وهمية ولا خلط للعملات.

### 9.3 صفحة العملاء

سطح المكتب:

- جدول كثيف.
- فلاتر الحالة والخطة والاشتراك.
- فرز وتصفح صفحات.
- لوحة تفاصيل جانبية.

الهاتف:

- صفوف بطاقات مختصرة.
- صفحة تفاصيل كاملة الشاشة.

تبويبات العميل:

- الملخص.
- الحساب والمساحة.
- الاشتراك.
- المدفوعات.
- الرسائل.
- البيانات المالية للقراءة فقط.
- سجل القرارات.

### 9.4 العنصر المميز

«سجل تحكم العميل»: خط زمني واحد يدمج إنشاء الحساب، تغييرات الاشتراك، مراجعات الدفع، الرسائل، الإيقاف، والوصول المالي.

## 10. النظام البصري

- الخط: IBM Plex Sans Arabic.
- المجال اللوني:
  - كحلي/حبر للتنقل والهوية.
  - ورق أبيض وcanvas هادئ للمحتوى.
  - أخضر للتفعيل والنجاح.
  - كهرماني للمخاطر والانتظار.
  - أحمر للإيقاف والرفض.
  - نيلي للإجراءات الأساسية.
- العمق: حدود هادئة وظلال خفيفة فقط.
- نصف القطر: صغير للأزرار، متوسط للبطاقات واللوحات.
- اللون يحمل معنى ولا يستخدم للزينة.
- حالات loading/empty/error/disabled إلزامية لكل سطح.

## 11. معالجة الأخطاء

- رسائل عربية من `getUserErrorMessage`.
- Mutation واحدة لا تعطل كل الصفحة.
- عند فشل إنشاء العميل:
  - تعرض مرحلة الفشل.
  - تلغي نية الإعداد إن لم تُستهلك.
  - تسجل فشل العملية دون كلمة المرور.
- لا تظهر كلمات المرور في logs أو audit أو toast.
- كلمة المرور المؤقتة تظهر مرة واحدة فقط ولا تخزن محليًا.
- عمليات التجديد تستخدم idempotency key ثابتًا لكل محاولة.

## 12. الاختبارات المطلوبة

### قاعدة البيانات

- غير المدير لا يستطيع استدعاء أي RPC إداري.
- انتقالات الاشتراك الصحيحة والخاطئة.
- التجديد idempotent.
- أرشفة خطة مستخدمة لا تكسر الاشتراكات.
- سجل before/after لكل إجراء.
- RPCs المالية للمدير قراءة فقط.
- المستخدم العادي لا يرى Views المدير.

### Edge Function

- رفض JWT غير صالح.
- رفض مستخدم غير مدير أو مدير موقوف.
- الدعوة.
- الإنشاء بكلمة مؤقتة.
- التعويض عند فشل ضبط الاشتراك.
- عدم تسريب service role أو كلمة المرور.

### Frontend

- إنشاء العميل بالطريقتين.
- تغيير الخطة والتجديد والحالات.
- الفلاتر وpagination.
- إرسال إشعار.
- عرض مالي قراءة فقط.
- تأكيدات الإجراءات وملاحظاتها.
- حالات التحميل والفشل والفراغ.

### E2E

- مدير ينشئ عميلًا ويجده في القائمة.
- العميل يعيّن كلمة المرور ويدخل.
- المدير يجدد الاشتراك ويظهر الحدث.
- مراجعة دفعة تحدث الاشتراك.
- غير المدير يُمنع من جميع صفحات المدير.
- اختبارات axe للهاتف وسطح المكتب.

## 13. مراحل التسليم

1. ترحيل التحكم الإداري + الاختبارات الأمنية.
2. Edge Function وإنشاء العملاء.
3. API/hooks مشتركة للوحة المدير.
4. جدول العملاء ولوحة التفاصيل.
5. الاشتراكات والخطط والمدفوعات.
6. الرسائل والتحليلات وسجل التدقيق.
7. القراءة المالية المقيدة.
8. E2E كامل وتدقيق advisors قبل النشر.

## 14. معايير القبول

- لا عملية إدارية حساسة تعتمد على إخفاء زر في الواجهة فقط.
- لا service key في المتصفح.
- كل تغيير اشتراك أو حساب أو خطة موثق.
- المدير يستطيع إنشاء عميل وتجديده وإدارة خطته دون Supabase Dashboard.
- كل الأرقام التشغيلية مشتقة من بيانات حقيقية.
- الهاتف وسطح المكتب قابلان للاستخدام دون تمرير أفقي إجباري.
- البناء وTypeScript وLint واختبارات الوحدة وE2E ناجحة.
